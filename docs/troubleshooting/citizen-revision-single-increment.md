# Изменение жителя «не сохраняется» (симуляция откатывается)

## Симптом

- В игровом тесте житель во время `simulateCitizen` уже имеет нужное состояние
  (видно по логу), но `inspectCitizen` сразу после возвращает **старое** состояние
  (например, голод снова 1000 и задачи нет — как будто только наняли).
- Внешне: тест на «голод стал критическим» падает; при этом побочные эффекты
  (например, изъятие предмета со склада) всё равно произошли — предмет пропал, а
  сытость не восстановилась.

## Причина

`SavedDataCitizenRepository.update(citizen, expectedRevision)` принимает запись только
если ревизия выросла **ровно на 1**:
```java
if (existing.revision() != expectedRevision
        || citizen.revision() != expectedRevision + 1) {
    return REVISION_CONFLICT; // молча отклонено
}
```
`CitizenAggregate.withSimulation(...)` (и `withProfession`, `reconcileEntity`, `recordDeath`)
каждый раз увеличивает ревизию на 1. Если за **один** шаг симуляции вызвать `withSimulation`
**дважды** (например, сначала `advance()` спадом нужд, потом отдельно «покормить»), ревизия
вырастет на 2 — и `update(..., original.revision())` будет отклонён. Изменение не сохранится,
но любые уже сделанные побочные эффекты (изъятие ресурса) останутся → рассинхрон.

> Тест может проходить **случайно**: если ассерт вида `hunger > 200` истинен и для
> несохранённого состояния (голод 1000 > 200), баг маскируется.

## Решение

Один персист = одно увеличение ревизии. Все правки жителя за один тик сводите в **один**
`withSimulation`, применённый к **сохранённому** (исходному) агрегату:

```java
CitizenAggregate advanced = simulationStep.advance(citizen, ...);        // R -> R+1 (в памяти)
CitizenAggregate result   = feedFromStorageIfRequested(citizen, advanced, ...); // R -> R+1 от ИСХОДНОГО
if (result != citizen) {
    citizenRepository.update(result, citizen.revision());                // R+1 == R+1, принято
}
```
Ключ: вторую правку выводим `original.withSimulation(finalNeeds, finalTask, ...)` — от
исходного `citizen` (ревизия R), а не от уже поднятого `advanced` (R+1). Промежуточные
значения берём из `advanced`, но базой ревизии остаётся сохранённый житель.

## Профилактика

- На один вызов `citizenRepository.update` — ровно один `withSimulation`/мутатор от того
  агрегата, чью `revision()` передаёте как `expectedRevision`.
- Не полагайтесь на ассерты, которые проходят и для несохранённого состояния
  (`> 200` при исходных 1000). Проверяйте точную ожидаемую величину.
