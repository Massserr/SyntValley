# SyntValley — спецификация проекта

Статус: архитектурная основа до начала реализации  
Целевая платформа: Minecraft 1.21.1, NeoForge 1.21.1, Java 21  
Mod id: `syntvalley`  
Базовый Java package: `dev.syntvalley`

## Архитектурная позиция

SyntValley — не симулятор колонии и не обёртка над vanilla villagers. Это мод о живой деревне: жители имеют собственные потребности, работу, память, характер и планы, а игрок становится участником её жизни, не единственным источником каждого решения.

Жители реализуются как собственные сущности. Их поведение строится на hybrid AI:

- Java выполняет частую, детерминированную и опасную для мира работу: навигацию, инвентари, ресурсы, задачи, безопасность, сохранения и синхронизацию;
- LLM редко и асинхронно предлагает намерения, объяснения, реплики, воспоминания и долгосрочные приоритеты;
- любой вывод LLM рассматривается как недоверенный ввод, проходит whitelist protocol и server-side validation;
- при недоступности LLM деревня продолжает работать по детерминированным правилам;
- разработка идёт постоянными vertical slices, каждый из которых является ранней частью итоговой архитектуры, а не временным прототипом.

## Видение продукта

Игрок ставит Synt Core и учреждает деревню либо присоединяет к нему нанятых жителей. Дальше игрок наблюдает и поддерживает поселение: обеспечивает ресурсы, разговаривает с конкретными жителями, рассматривает их предложения, помогает с опасностями и при необходимости управляет приоритетами в creative mode.

Ожидаемый эффект — не абсолютная автономность и не иллюзия всемогущего ИИ, а правдоподобная последовательность:

1. У жителей есть понятные текущие потребности.
2. Деревня обнаруживает дефицит или возможность.
3. Конкретный житель формирует объяснимое намерение.
4. Java проверяет, можно ли превратить намерение в проект или задачи.
5. Жители постепенно выполняют работу в загруженных чанках.
6. Результат меняет память, настроение, ресурсы и будущие решения.

## Цели

### Игровые

- Создать ощущение деревни, которая живёт без постоянного микроменеджмента игрока.
- Сделать жителей различимыми благодаря личности, настроению, привычкам, профессии и памяти.
- Дать деревне способность объяснимо реагировать на голод, дефициты, опасность, смерть, повреждения и новые возможности.
- Позволить жителям предлагать проекты и менять приоритеты через обсуждение, но сохранить правила Minecraft и контроль Java.
- Сохранить ванильную визуальную читаемость на раннем этапе: знакомые пропорции, простые модели и текстуры, без преждевременной дорогой арт-системы.
- Развести личный разговор, обзор деревни, административное управление и отладку по разным интерфейсам.

### Технические

- Работать на NeoForge 1.21.1 и Java 21.
- С первого среза соблюдать server-authoritative модель, в том числе в singleplayer.
- Не блокировать Minecraft server thread HTTP-запросами, парсингом больших ответов или файловым вводом-выводом.
- Использовать ограниченные очереди, тайм-ауты, квоты и backpressure для LLM.
- Сохранять стабильные идентификаторы деревень, жителей, задач и проектов между перезапусками.
- Отделить persistent domain state от transient runtime state.
- Заложить schema versioning, последовательные миграции и безопасную реакцию на неизвестную будущую схему.
- Ограничить работу симуляции tick budget и не допускать модели «каждый житель вызывает LLM каждый tick».
- Позволить заменить Ollama другим backend через порт `LlmBackend`, не меняя доменную модель и action protocol.
- Сохранить seam для будущей упрощённой offline simulation, не обещая её в первых версиях.

## Основные понятия

| Понятие | Значение |
|---|---|
| Village | Доменный агрегат деревни: идентичность, жители, политика, память, проекты и сводные ресурсы. |
| Synt Core | Физический якорь Village в мире. Хранит ссылку на `VillageId`, но не является единственным хранилищем деревни. |
| Village Console | Отдельный блок для Overview, журнала и административных экранов. |
| Citizen | Собственный AI-житель с постоянным `CitizenId`; его сущность является world-presence доменного объекта. |
| Intent | Высокоуровневое намерение, которое может предложить LLM или deterministic planner. |
| Task | Исполнимая Java-машина состояний с условиями, резервированием ресурсов и результатом. |
| Project | Долгоживущая цель деревни, разложенная Java на задачи; строительство использует шаблон. |
| Memory | Ограниченная структурированная запись о событии, значимости и участниках, а не бесконечная история чата. |
| Decision Log | Аудит: какой снимок рассматривался, что было предложено, принято или отклонено и почему. |
| Action Protocol | Версионированный whitelist семантических предложений от LLM к Java. |

## Границы первой продуктовой дуги

### Входит

- Synt Core и создание/восстановление идентичности деревни.
- Отдельный Village Console для обзора и управления.
- Найм или административное создание собственных жителей без vanilla breeding.
- Собственная entity жителя с vanilla-подобным ранним визуальным стилем.
- Потребности: голод, отдых, безопасность, инструменты, социальность и ощущение цели.
- Личность, настроение и небольшое число привычек с детерминированным влиянием на оценки.
- Профессии как data-driven определения и назначение навыков/рабочих задач.
- Задачи, планировщик, резервирование ресурсов и ограниченное исполнение в loaded chunks.
- Индивидуальная и деревенская память с лимитами, дедупликацией и угасанием значимости.
- Шаблонные дом, ферма, склад и мастерская; Java выбирает и проверяет размещение.
- Проекты жителей: создание предложения, проверка, принятие, исполнение и журналирование.
- Ollama backend с начальной моделью `qwen3:8b`, structured output и возможностью замены backend.
- Редкие citizen-thinking и ещё более редкие village-planning вызовы.
- Личный текстовый чат только при взаимодействии с конкретным жителем.
- Village Overview, creative-only Priority Management, Memory/Decision Log и Debug screens.
- Server-authoritative networking с request/response и ограниченными snapshots/deltas.
- Deferred dirty tracking, сохранение на штатных циклах мира, миграции и recovery policy.
- Детерминированный fallback без LLM.
- Unit, serialization, protocol, Game Tests и dedicated-server smoke tests.

### Не входит в первые версии

- Голосовой чат и распознавание/синтез речи.
- Семьи, дети, сложная романтика и размножение.
- Рейды как отдельная система SyntValley.
- Произвольное генеративное строительство из блоков, придуманных LLM.
- Выполнение LLM-команд вроде `place_block`, `break_block`, `move_to`, `attack` или server command.
- Полноценная симуляция entity/pathfinding в выгруженных чанках.
- Принудительная загрузка чанков самим модом; совместимость с внешним chunk loader допустима.
- Копирование кода, package layout, class names или файловой архитектуры MineColonies/AIMineColonies.
- Полная экономическая модель, политическая фракционная система и обещание человеческого уровня общения.
- Неограниченная долговременная память или хранение полного LLM transcript в save.

## Core gameplay loop

### Учреждение

1. Игрок размещает Synt Core в допустимом месте.
2. Server создаёт `VillageId`, доменную запись и привязывает к ней core position/dimension.
3. Игрок размещает или привязывает Village Console.
4. Игрок нанимает первого жителя предусмотренным рецептом/контрактом либо использует creative-инструмент разработки.

### Ежедневная жизнь

1. Java обновляет потребности на разреженных cadence, а не все системы каждый tick.
2. Планировщик оценивает доступные задачи, приоритеты, профессию, путь, инструменты и резервации.
3. Житель выполняет ограниченный шаг задачи: перемещение, получение предмета, работа или отдых.
4. События работы меняют ресурсы и формируют краткие структурированные воспоминания.
5. UI получает server-approved read model, но не доменные объекты напрямую.

### Возникновение инициативы

1. Event/threshold detector отмечает значимое состояние: длительный дефицит, простой, риск или возможность.
2. Decision scheduler применяет cooldown, квоты и проверяет, нужен ли вообще LLM.
3. В async backend уходит ограниченный immutable snapshot без доступа к объектам Minecraft.
4. Ответ парсится по версии protocol и возвращается на server thread.
5. Java валидирует предложение, принимает его, отклоняет или заменяет deterministic fallback.
6. Принятое намерение становится goal/project/task только через application service.

### Участие игрока

Игрок помогает ресурсами, разговаривает, рассматривает журнал причин, принимает запросы, наблюдает последствия и в creative mode может менять стратегические веса. Survival-игрок не получает административный bypass, а клиент никогда не меняет состояние напрямую.

## Player experience

### Требуемое ощущение

- Житель должен выглядеть занятым понятной работой, а не хаотично блуждать.
- Действия должны иметь наблюдаемую причину: голодный житель прекращает необязательную работу; дефицит инструмента создаёт запрос; опасность вытесняет обычные приоритеты.
- Различия личности должны быть статистически заметны за время, но не превращать каждое действие в случайность.
- LLM-реплика не считается истинной, пока факты не взяты из server snapshot; выдуманные факты не попадают в каноническую память.
- Отказ или ошибка LLM не должны замораживать жителя, деревню или UI.
- Журнал решений должен позволять разработчику понять «почему», не показывая скрытые reasoning traces модели.

### UI ownership

| Интерфейс | Как открывается | Назначение |
|---|---|---|
| Citizen Chat | Server-approved interaction с entity | Личный диалог и конкретные просьбы. |
| Village Overview | Village Console | Состав, нужды, ресурсы, проекты и предупреждения. |
| Priority Management | Вкладка Console; только creative/permission | Изменение ограниченных весов и политик. |
| Memory / Decision Log | Village Console или debug permission | Структурированные воспоминания, предложения, отказы и причины. |
| Debug Screen | Dev/OP permission | Очереди, LLM latency, budgets, path/task state, dirty state. |

## Инварианты продукта

1. Только logical server создаёт или изменяет каноническое состояние.
2. `CitizenId` и `VillageId` не выводятся из entity id, имени, позиции или chunk.
3. Асинхронный поток не хранит и не изменяет `Level`, `Entity`, `BlockEntity`, registry object или mutable domain object.
4. LLM-ответ не вызывает world mutation напрямую.
5. Низкоуровневая работа существует только как Java task step с повторной проверкой в момент исполнения.
6. Каждая очередь, коллекция памяти, журнал и LLM-вызов имеют явный предел.
7. Не загруженный chunk означает pause/alternative task, но не неявную force-load операцию.
8. Отсутствие LLM означает снижение разнообразия, а не остановку базовой симуляции.
9. Core/Console block entity содержит binding и локальные UI-данные; канонический агрегат деревни хранится world-scoped.
10. Смена backend не меняет action protocol и доменные команды.

## Критерии качества

### Производительность

- Ни один HTTP-вызов не выполняется на server thread.
- Системы симуляции имеют cadence и распределяются по tick’ам; нет полного обхода всех жителей всеми подсистемами каждый tick.
- Tick budget измеряется и виден в debug screen; при перегрузке необязательные расчёты откладываются.
- LLM queue bounded; отказ постановки в очередь является штатным наблюдаемым результатом.
- Memory/decision records имеют лимиты по количеству и длине строк.

### Надёжность

- Save/load round-trip сохраняет идентичность и не дублирует жителей.
- Неизвестная более новая save schema не загружается с молчаливой потерей данных.
- Повторная доставка LLM-ответа не создаёт вторую задачу: decision id обрабатывается идемпотентно.
- Удаление core не уничтожает деревню немедленно; она переходит в recoverable lifecycle state.
- Ошибки одного жителя, проекта или backend не обрывают server tick.

### Multiplayer readiness

- Dedicated server загружается без `net.minecraft.client` linkage.
- Все client requests проверяют sender, distance/context, permission, identifiers, revision и payload bounds.
- UI открывается сервером либо подтверждается сервером; DTO минимальны и не раскрывают mutable internals.
- Creative-only управление повторно проверяется на logical server.

## Конфигурация по категориям

Точные defaults калибруются профилированием и не являются частью save schema.

- `common`: cadence симуляции, лимиты памяти, включение UI/debug-функций.
- `server`: LLM enabled/backend/base URL/model, executor size, queue capacity, timeout, cooldowns, budgets, action-policy limits.
- `client`: визуальные настройки и объём отображаемого журнала; не влияет на правила мира.
- Секреты и произвольные auth headers не должны попадать в world save, network DTO или обычный лог.

## Риски продукта

| Риск | Последствие | Основная мера |
|---|---|---|
| Нестабильный вывод `qwen3:8b` | Невалидные/странные решения | Structured output, строгий parser, semantic validation, fallback. |
| Большое число жителей | Tick lag и pathfinding spikes | Cadence, budgets, staggered scheduler, лимиты активных задач. |
| Длинная память/промпты | Рост latency и RAM | Bounded records, summarization as suggestion, token/character budgets. |
| Ошибка save/migration | Потеря деревни | Versioned root, backups штатного мира, последовательные миграции, fail closed. |
| Client spoofing | Административные изменения | Server authority, permission/context/revision validation. |
| Prompt injection через игрока | Обход action rules | Player text как quoted data, immutable system contract, whitelist validation. |
| Недоступный Ollama | Потеря «живости» | Circuit breaker, cooldown и deterministic planner/dialogue fallback. |
| Строительство портит мир | Потеря блоков/конфликты | Только шаблоны, protected-area checks, resource reservation, staged execution. |
| Расхождение entity/domain state | Дубли или «призраки» | Stable IDs, binding reconciliation, explicit lifecycle states. |
| Слишком случайный характер | Нечитаемое поведение | Личность меняет score в ограниченных пределах; safety/needs остаются жёсткими. |

Подробный реестр рисков и принятых решений находится в `RISKS_AND_DECISIONS.md`.

## Метрики успеха

### Технические

- Build, unit tests и Game Tests воспроизводимо проходят на чистом checkout.
- Dedicated-server smoke test проходит без клиента и без запущенного Ollama.
- При выключенном/зависшем backend TPS не блокируется, очередь ограничена, fallback наблюдаем.
- После save/restart все стабильные идентификаторы, активные задачи и проекты либо восстановлены, либо переведены в документированное recovery state.
- Отклонённые protocol actions имеют machine-readable reason code и не меняют мир.

### Игровые

- Деревня с несколькими жителями поддерживает базовый цикл «потребность → задача → ресурс → память» без ручной команды на каждый шаг.
- Два жителя с разными traits при одинаковом контексте могут выбирать разные необязательные действия, но одинаково соблюдают безопасность.
- Игрок может из UI объяснить текущий приоритет, активную задачу и причину последнего значимого решения.
- Проект из разрешённого шаблона проходит стадии proposal, validation, reservation, build и completion.
